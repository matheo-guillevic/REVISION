Problème I - Partie cours

1.1 En VHDL, quel type normalisé doit être utilisé pour réaliser des opérations arithmétiques sur des vecteurs binaires signés en complément à 2 [2] ? Montrez en détaillant l'opération sur des nombres binaires signés en « complément à 2 » comment le résultat de l'opération « -6 x 7 » est obtenu [6].

Correction et raisonnement

Étape 1 : Choix du type VHDL.
Le type normalisé à utiliser pour traiter des valeurs signées est SIGNED (issu de la bibliothèque ieee.numeric_std).

Étape 2 : Préparation des opérandes.
On code les valeurs sur 4 bits en appliquant la convention du complément à 2 :

  • 7 s'écrit en binaire : \(0111_2\)
  • -6 s'obtient en prenant l'inverse de 6 (\(0110_2 \rightarrow 1001_2\)) et en ajoutant 1 : \(1010_2\)

Étape 3 : Extension de signe et multiplication.
Pour que les additions soient correctes lors de la multiplication, on étend le signe du nombre négatif sur 8 bits (\(1010_2 \rightarrow 11111010_2\)). On multiplie ensuite par \(0111_2\) : \[11111010_2 \times 0111_2 = \mathbf{11010110_2}\] Ce résultat correspond bien à la valeur -42 en complément à 2 sur 8 bits.

1.2 Dessinez le chronogramme du signal S suivant [4] :

S <= '0', '1' after 10 ns, '0' after 25 ns, '1' after 30 ns;
Correction et raisonnement

Étape 1 : Analyse de l'instruction temporelle.
Le mot clé after génère des événements dont les délais sont absolus par rapport à l'instant d'évaluation initial (\(t=0\)).

Étape 2 : Tracé des événements.
La ligne de temps du signal S sera la suivante :

  • À \(t = 0\text{ ns}\) : Le signal s'initialise à '0'.
  • À \(t = 10\text{ ns}\) : Le signal passe à '1'.
  • À \(t = 25\text{ ns}\) : Le signal retombe à '0'.
  • À \(t = 30\text{ ns}\) : Le signal remonte à '1' indéfiniment.

1.3 Que se passe-t-il si on met l'instruction « wait ; » dans un processus VHDL [2] ? Au contraire, que se passe-t-il si on ne met aucun « wait » dans un processus VHDL [2] ?

Correction et raisonnement

Étape 1 : Comportement avec wait ;.
L'instruction wait ; provoque un blocage inconditionnel et infini. Le processus s'exécutera une unique fois lors de l'initialisation de la simulation, puis se mettra en pause pour toujours.

Étape 2 : Comportement sans aucun wait.
Si un processus n'a ni wait à l'intérieur de son bloc, ni liste de sensibilité dans sa déclaration, il reboucle instantanément sur lui-même à la fin de son exécution. Cela crée une boucle infinie dans le même cycle delta, ce qui fige (crashe) le simulateur.

1.4 En VHDL, est-ce que l'ordre des processus dans le code a une importance [2] ? Si non, dans quel ordre sont exécutés les processus [4] ?

Correction et raisonnement

Étape 1 : Importance de l'ordre structurel.
Non, l'ordre d'écriture physique des processus dans le fichier n'a absolument aucune importance. Le VHDL modélise du matériel physique fonctionnant en parallèle (exécution concurrente).

Étape 2 : Mécanisme réel d'exécution.
Les processus sont exécutés par le moteur de simulation de façon événementielle. Un processus ne s'exécute que lorsqu'un des signaux présents dans sa liste de sensibilité (ou dans ses instructions wait) subit une transition (événement).

1.5 Donnez des codes VHDL réalisant les mêmes fonctions que les 4 codes Verilog suivants [14=2+3+3+6] :

assign x = a & b & c;
 
always @(*) begin
  case (ad)
    2'b00: s = e0;
    2'b01: s = e1;
    2'b10: s = e2;
    default: s = e3;
  endcase
end
Correction et raisonnement

Étape 1 : Traduction logique continue.
Le assign combinatoire Verilog devient une simple affectation concurrente VHDL. \[\mathbf{\texttt{x <= a and b and c;}}\]
Correction et raisonnement

Étape 1 : Traduction du Multiplexeur.
Un case combinatoire se traduit élégamment en VHDL par une affectation avec sélection (with ... select).

with ad select s <= 
    e0 when "00", 
    e1 when "01", 
    e2 when "10", 
    e3 when others;
 
always @(*)
begin
  if (c) begin
    q = d;
  end
end
 
module SISO_SIPO (
  input wire clk, input wire d,
  output wire s, output wire [4:1] q
);
reg [4:1] q_temp;
always @(posedge clk) begin
  q_temp <= {q_temp[3:1], d};
end
assign q = q_temp;
assign s = q_temp[4];
endmodule
Correction et raisonnement

Étape 1 : Traduction du Latch.
L'absence de else génère un Latch. En VHDL, on utilise un process standard.

process(c, d) begin
  if c = '1' then
    q <= d;
  end if;
end process;
Correction et raisonnement

Étape 1 : Registre à décalage synchrone.
L'opérateur de concaténation Verilog \texttt{\{\}} devient & en VHDL. Un signal interne q_temp est nécessaire pour l'affectation.

process(clk) begin
  if rising_edge(clk) then
    q_temp <= q_temp(3 downto 1) & d;
  end if;
end process;
q <= q_temp;
s <= q_temp(4);

Partie A: Analyse d'une description VHDL - 50 pts

2.1 Le reset utilisé dans l'architecture rtl ci-dessous est-il synchrone ou asynchrone ? [2]

Correction et raisonnement

Étape 1 : Analyse de la priorité.
Dans le code fourni, la condition if (reset='1') est placée au tout début du processus, avant l'évaluation de l'horloge.

Étape 2 : Conclusion matérielle.
Puisque le changement d'état du reset est appliqué immédiatement sans attendre le front de l'horloge (qui est placé dans le elsif), le reset est de type asynchrone.

2.2 Modifiez le code VHDL ci-dessous pour le rendre synchrone (s'il est asynchrone) ou asynchrone (s'il est synchrone) [2].

Correction et raisonnement

Étape 1 : Principe de la synchronisation.
Pour forcer un comportement synchrone, le signal de reset ne doit être évalué qu'au moment précis d'un front d'horloge.

Étape 2 : Modification du code VHDL.
On place le test du reset à l'intérieur de la condition rising_edge(clk).

process(clk) 
begin
  if rising_edge(clk) then
    if (reset='1') then
      r <= (others=>'0');
      buf <= '0';
    else
      -- suite du compteur (r <= r + 1, etc.)
    end if;
  end if;
end process;

2.3 Décrire les bibliothèques et l'entité nécessaires à cette architecture [2+2].

Correction et raisonnement

Étape 1 : Déduction des bibliothèques.
L'utilisation de std_logic requiert ieee.std_logic_1164.all. Les opérations arithmétiques requièrent ieee.numeric_std.all.

Étape 2 : Définition de l'entité.
On identifie les entrées (clk, reset, w) et les sorties (pulse).

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity hidden is
  port (
    clk, reset : in std_logic;
    w          : in std_logic_vector(3 downto 0);
    pulse      : out std_logic
  );
end hidden;

2.4 Décrire en VHDL un testbench permettant d'exercer le fonctionnement de cette entité pour 2 valeurs successives différentes de w [16].

Correction et raisonnement

Étape 1 : Architecture du banc d'essai.
Il faut instancier le composant (DUT), générer une horloge, et créer un processus de stimulus séquentiel.

Étape 2 : Processus de test (Stimuli).

-- (Instanciation et horloge clk <= not clk after 5 ns; omises pour brievete)
process begin
  reset <= '1'; wait for 15 ns; reset <= '0';
  
  w <= "0011"; -- Premier test (w=3)
  wait for 160 ns; -- Attente de 16 cycles complets (16x10ns)
  
  w <= "0110"; -- Second test (w=6)
  wait for 160 ns;
  
  wait;
end process;

2.5 Faire un chronogramme décrivant l'évolution des signaux d'entrée et de sortie, ainsi que de la valeur de r, lors de la simulation du testbench précédent [16].

Correction et raisonnement

Étape 1 : Comportement du compteur.
La variable interne r est un compteur sur 4 bits qui s'incrémente à chaque cycle, formant une "dent de scie" de 0 à 15 de manière continue.

Étape 2 : Comportement de la sortie pulse.
La sortie pulse vaut '1' tant que r < w.

  • Si w=3, le signal pulse est à l'état haut pendant 3 cycles, puis à l'état bas pendant 13 cycles.
  • Si w=6, il est à l'état haut pendant 6 cycles, puis bas pendant 10 cycles.

2.6 Combien de bascules seront créées par cette architecture rtl lors de sa synthèse. Justifiez [4].

Correction et raisonnement

Étape 1 : Règles d'inférence de bascules.
Une bascule D est synthétisée pour chaque bit d'un signal assigné de manière conditionnelle à l'intérieur d'un bloc rising_edge(clk).

Étape 2 : Comptage des ressources.
Le signal r est un vecteur de 4 bits mis à jour sous horloge \(\rightarrow\) 4 bascules. Le signal buf est sur 1 bit, également mis à jour sous horloge \(\rightarrow\) 1 bascule. La sortie pulse est combinatoire pure (reliée à buf). \[\mathbf{Total = 5 \text{ bascules}}\]

2.7 En analysant le code VHDL de l'architecture rtl et le résultat de la simulation de votre testbench, expliquez quelle est la fonction de cette architecture [2]. Expliquez comment varie la forme du signal pulse en fonction de la valeur de w [4].

Correction et raisonnement

Étape 1 : Identification du composant.
Le code décrit un générateur de signal MLI (Modulation de Largeur d'Impulsion) ou PWM.

Étape 2 : Effet de l'entrée W.
Le signal d'entrée w contrôle le rapport cyclique de la sortie. Plus la valeur numérique de w est grande, plus le temps passé à l'état haut par le signal pulse augmente proportionnellement sur une période fixe de 16 cycles d'horloge.

Partie B: Ecriture d'une machine à états - 34 pts

3.1 Dessinez le diagramme de transitions d'une FSM de Moore qui a deux entrées X et rst et une sortie Y. Chaque fois que X change de 0 à 1, Y passe à 1 pour deux cycles d'horloge et ensuite revient à 0 -- même si X est toujours 1 [12].

Correction et raisonnement

Étape 1 : Définition des états nécessaires.
Pour garantir que la sortie ne s'active que pendant 2 cycles et ignore un X maintenu à 1, 4 états sont requis : Repos (S0), Temporisation 1 (S1), Temporisation 2 (S2), et Attente/Verrouillage (S3).

Étape 2 : Définition du Graphe.

  • S0 (\(Y=0\)) : Si X=1 \(\rightarrow\) Aller vers S1. Sinon \(\rightarrow\) Rester en S0.
  • S1 (\(Y=1\)) : Transition inconditionnelle vers S2 au prochain cycle.
  • S2 (\(Y=1\)) : Si X=1 \(\rightarrow\) Aller vers S3 (pour se verrouiller). Si X=0 \(\rightarrow\) Retour S0.
  • S3 (\(Y=0\)) : Attendre. Si X=0 \(\rightarrow\) Retour S0. Sinon, rester en S3.

3.2 Faire un chronogramme représentant l'horloge clk, l'entrée X, l'entrée rst et la sortie Y [6].

Correction et raisonnement

Étape 1 : Comportement de la machine de Moore.
Par nature (Moore), la sortie est synchronisée sur les états. Il y aura donc un décalage d'un front d'horloge entre le passage de X à 1 et le passage de Y à 1.

Étape 2 : Visualisation temporelle.
Au front d'horloge n°1, la machine détecte X=1 et passe en S1. Au front n°2, Y vaut 1. Au front n°3, la machine passe en S2 (Y reste à 1). Au front n°4, la machine passe en S3 (Verrou), et Y retombe à 0, même si X est toujours haut.

3.3 Ecrire le code VHDL de cette FSM (entité et architecture complète) [12]

Correction et raisonnement

Étape 1 : Architecture VHDL à deux processus.
On code la FSM en séparant clairement le registre d'état (synchrone) de la logique de calcul du prochain état (combinatoire).

Étape 2 : Code complet.

architecture rtl of edge_fsm is
  type state_t is (S0, S1, S2, S3);
  signal state, next_state : state_t;
begin
  process(clk) begin -- Processus Synchrone
    if rising_edge(clk) then
      if rst = '1' then state <= S0;
      else state <= next_state; end if;
    end if;
  end process;

  process(state, X) begin -- Processus Combinatoire
    next_state <= state; Y <= '0';            
    case state is
      when S0 => if X = '1' then next_state <= S1; end if;
      when S1 => Y <= '1'; next_state <= S2;
      when S2 => Y <= '1';
                 if X = '1' then next_state <= S3; 
                 else next_state <= S0; end if;
      when S3 => if X = '0' then next_state <= S0; end if;
    end case;
  end process;
end architecture;

3.4 Combien au minimum de bascules seront nécessaires pour implémenter votre description précédente et la FSM correspondante [4] ?

Correction et raisonnement

Étape 1 : Dénombrement des états.
Le système modélisé contient 4 états distincts (S0, S1, S2, S3).

Étape 2 : Calcul des bascules matérielles.
Pour coder \(N\) états en binaire pur, le nombre minimum de bascules \(n\) nécessaires répond à la formule : \(n = \lceil \log_2(N) \rceil\). \[\mathbf{\lceil \log_2(4) \rceil = 2 \text{ bascules}}\]
