Probleme 1

Problème I - Conception et validation d'un « timer »

Contexte

L'objectif est de concevoir un timer générant périodiquement une impulsion d'une période d'horloge de large. Ce timer est contrôlé par une horloge interne de période \(1\,\mu\text{s}\). Il a une largeur de 32 bits.

Lecture du Schéma : Liste des éléments et leur utilité

Le schéma illustre l'architecture matérielle (chemin de données et contrôle) du timer. Voici comment le lire pas-à-pas :

  • 1 microsec oscillator (Oscillateur) : Fournit le signal d'horloge de base (période de \(1\,\mu\text{s}\)) qui cadence tous les éléments synchrones du circuit.
  • Bloc "-1" (Décrémenteur combinatoire) : Réalise une opération de soustraction de 1 sur l'entrée \(M\). Cela est nécessaire car le compteur décrémente jusqu'à 0 inclus. Pour compter \(M\) cycles, il faut démarrer le comptage à \(M-1\).
  • 32-bit register (Registre de mémorisation) : Sauvegarde la valeur \(M-1\) à chaque fois que la commande load est active (à '1'). Il sert de mémoire pour conserver la configuration du timer de manière autonome.
  • 4-bit 2x1 Mux (Multiplexeur 2 vers 1) : Aiguille la donnée qui sera chargée dans le compteur. Si load est à '1', il laisse passer la nouvelle valeur \(M-1\). Si load est à '0', il laisse passer la valeur sauvegardée dans le registre.
  • Porte logique OU (OR gate) : Génère le signal de chargement (ld) du compteur (si load = 1 ou tc = 1).
  • 32-bit down-counter (Décompteur 32 bits) : C'est le cœur du timer. Activé par enable, il décrémente sa valeur à chaque front d'horloge. Lorsqu'il atteint 0, sa sortie tc passe à 1.

Question 1 : Quelle valeur doit être programmée (c'est-à-dire envoyée vers M) pour générer une impulsion toutes les 300 ms ?

Correction et raisonnement

Étape 1 : Analyse des périodes.
L'horloge a une période de \(1\,\mu\text{s}\). On souhaite une période globale de \(300\text{ ms}\), ce qui équivaut à \(300\,000\,\mu\text{s}\).

Étape 2 : Calcul du nombre de cycles.
Il faut donc compter exactement \(300\,000\) cycles d'horloge entre chaque impulsion.

Étape 3 : Logique de comptage.
Le circuit décrémente de \(M-1\) jusqu'à \(0\) (soit un total de \(M\) états ou \(M\) cycles). Le bloc "-1" matériel s'occupe déjà de soustraire 1 à la valeur de consigne.

Étape 4 : Déduction de M.
La valeur à programmer sur l'entrée \(M\) correspond donc directement au nombre de cycles souhaités : \[\mathbf{M = 300\,000}\]

Question 2 : Faire un chronogramme pour deux impulsions avec \(M=3\) puis deux impulsions avec \(M=2\).

Correction et raisonnement

Étape 1 : Initialisation (Cycle 0).
load = 1, \(M = 3\). Le bloc "-1" calcule \(M-1 = 2\). Le registre R32 charge 2. Le Mux32 laisse passer 2. Le compteur C charge 2 car la porte OU donne 1.

Étape 2 : Première séquence d'impulsions (M=3).

  • Cycle 1 : load = 0, enable = 1. C = 2. tc (Q) = 0. Au front, C \(\rightarrow\) 1.
  • Cycle 2 : C = 1, Q = 0. Au front, C \(\rightarrow\) 0.
  • Cycle 3 (Impulsion) : C = 0. Donc Q = 1 (tc=1). La porte OU vaut 1. Le Mux32 laisse passer R32 (2). Au front, C recharge 2.
  • Cycles 4, 5, 6 : Identique aux cycles 1 à 3. On obtient une 2ème impulsion.

Étape 3 : Seconde séquence d'impulsions (M=2).

  • Cycle 7 (Changement) : load = 1, \(M = 2\). R32 et C chargent 1 (\(2-1\)).
  • Cycle 8 (Impulsion) : load = 0, enable = 1. C passe de 1 à 0 au cycle 9. La période est maintenant de 2 cycles.

Question 3 : Est-ce que les périodes des impulsions générées correspondent aux valeurs de M successives (3 puis 2) ? Si ce n'est pas le cas, expliquez.

Correction et raisonnement

Étape 1 : Vérification pour M=3.
Pour \(M=3\), le compteur prend les valeurs \(2 \rightarrow 1 \rightarrow 0\). Il faut donc 3 cycles d'horloge complets pour revenir à l'état initial. La période correspond.

Étape 2 : Vérification pour M=2.
Pour \(M=2\), le compteur prend les valeurs \(1 \rightarrow 0\). Il faut donc 2 cycles d'horloge. La période correspond.

Étape 3 : Conclusion et rôle du bloc soustracteur.
Oui, les périodes correspondent parfaitement. Le bloc "-1" placé en amont compense le fait que le compteur passe par l'état "0". Il garantit que la période mesurée en cycles d'horloge soit exactement égale à la valeur entière \(M\) demandée.

Question 4 : Donnez une description en VHDL de l'entité du timer.

Correction et raisonnement

Étape 1 : Identification des ports et bibliothèques.
Le schéma montre les entrées d'horloge, de contrôle (load, enable), le bus de données \(M\) sur 32 bits et la sortie scalaire \(Q\). On inclut numeric_std en prévision de l'architecture.

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity timer is
    port (
        clk    : in  std_logic;
        load   : in  std_logic;
        enable : in  std_logic;
        M      : in  std_logic_vector(31 downto 0);
        Q      : out std_logic
    );
end entity timer;

Question 5 : Décrire en VHDL RTL l'architecture rtl de ce circuit.

Correction et raisonnement

Étape 1 : Déclaration des signaux internes.
On déclare les signaux de type unsigned pour les bus de données (M32, R32, Mux32, C) pour permettre l'arithmétique.

Étape 2 : Implémentation combinatoire et processus synchrones.
La logique combinatoire gère le soustracteur, le multiplexeur, la porte OU et le comparateur à zéro. Deux processus distincts gèrent les éléments mémorisants (Registre R32 et Compteur C).

architecture rtl of timer is
    signal M32_uns, R32_uns, Mux32_uns, C_uns : unsigned(31 downto 0);
    signal OR1, Q_int : std_logic;
begin
    -- Logique Combinatoire
    M32_uns <= unsigned(M) - 1;
    Mux32_uns <= M32_uns when load = '1' else R32_uns;
    
    Q_int <= '1' when C_uns = to_unsigned(0, 32) else '0';
    OR1 <= load or Q_int;
    Q <= Q_int;

    -- Registre de Memorisation (R32)
    process(clk) begin
        if rising_edge(clk) then
            if load = '1' then
                R32_uns <= M32_uns;
            end if;
        end if;
    end process;

    -- Decompteur 32 bits (C)
    process(clk) begin
        if rising_edge(clk) then
            if OR1 = '1' then
                C_uns <= Mux32_uns;     -- Chargement synchrone
            elsif enable = '1' then
                C_uns <= C_uns - 1;     -- Decomptage synchrone
            end if;
        end if;
    end process;
end architecture rtl;

Question 6 : Commentez la fonction du code Verilog fourni. (Note: Fait référence à un bout de code Verilog issu du sujet original).

Correction et raisonnement

Étape 1 : Traduction du comportement.
Le code Verilog décrit un multiplexeur 2 vers 1 modélisé de manière combinatoire (sensibilité always @(*) ou always @(load or M32 or R32)).

Étape 2 : Conclusion.
Il aiguille le signal : si load vaut 1, la sortie prend la valeur de l'entrée M32. Si load vaut 0, la sortie prend la valeur de l'entrée R32.

Question 7 : Combien de bascules générera la synthèse de votre architecture précédente ? Justifiez.

Correction et raisonnement

Étape 1 : Analyse des processus synchrones.
Le circuit contient deux processus réagissant au front montant de l'horloge (rising_edge(clk)), qui vont donc synthétiser des bascules D.

Étape 2 : Dénombrement.

  • Le premier processus mémorise le vecteur R32_uns (32 bits) \(\rightarrow\) 32 bascules.
  • Le second processus mémorise le vecteur C_uns (32 bits) \(\rightarrow\) 32 bascules.

Tous les autres signaux (Mux, bloc -1, porte OU) sont modélisés en combinatoire. \[\mathbf{Total = 64 \text{ bascules}}\]

Question 8 : Donnez une description du testbench tb_timer permettant de retrouver exactement le même chronogramme que celui de la question 2.

Correction et raisonnement

Étape 1 : Instanciation et horloge.
On crée l'entité de test, on génère une horloge de \(1\,\mu\text{s}\) et on instancie le timer.

Étape 2 : Scénario (Stimuli).

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity tb_timer is
end entity tb_timer;

architecture sim of tb_timer is
    signal clk, load, enable, Q : std_logic := '0';
    signal M : std_logic_vector(31 downto 0) := (others => '0');
    constant CLK_PERIOD : time := 1 us;
begin
    UUT: entity work.timer port map(clk, load, enable, M, Q);

    clk_process : process begin
        clk <= '0'; wait for CLK_PERIOD / 2;
        clk <= '1'; wait for CLK_PERIOD / 2;
    end process;

    stim_proc : process begin
        -- Initialisation (Cycle 0)
        M <= std_logic_vector(to_unsigned(3, 32));
        load <= '1'; enable <= '0'; wait for CLK_PERIOD;
        
        -- Generation de 2 impulsions (M=3 -> 6 cycles)
        load <= '0'; enable <= '1'; wait for 6 * CLK_PERIOD;
        
        -- Changement (M=2)
        M <= std_logic_vector(to_unsigned(2, 32));
        load <= '1'; wait for CLK_PERIOD;
        
        -- Generation de 2 impulsions (M=2 -> 4 cycles)
        load <= '0'; wait for 4 * CLK_PERIOD;
        
        enable <= '0'; wait;
    end process;
end architecture sim;
Probleme 2

Problème II - Contrôleur de porte de garage

Contexte

Une porte de garage est contrôlée par une machine à états. ctr(1) gère l'alimentation du moteur (1=ON, 0=OFF) et ctr(0) définit le sens (0=ouverture, 1=fermeture). Les capteurs sen1 et sen2 détectent respectivement l'ouverture totale et la fermeture totale. Le bouton remt donne les ordres.

Question 2.1 : Donnez un diagramme de transitions de la machine à états de Moore avec remt sous forme d'impulsion d'une seule période.

Correction et raisonnement

Étape 1 : Définition des états et sorties.
La sortie dépend uniquement de l'état (Machine de Moore). On définit les vecteurs ctr associés :

  • FERME (ctr = "0X") : Moteur coupé.
  • OUVERTURE (ctr = "10") : Moteur ON, sens ouverture.
  • OUVERT (ctr = "0X") : Moteur coupé.
  • FERMETURE (ctr = "11") : Moteur ON, sens fermeture.
  • STOP_OUVERTURE (ctr = "0X") : Arrêt d'urgence/pause.
  • STOP_FERMETURE (ctr = "0X") : Arrêt d'urgence/pause.

Étape 2 : Conditions de transition.

  • Depuis FERME : Si remt = 1 \(\rightarrow\) OUVERTURE.
  • Depuis OUVERTURE : Si sen1 = 1 \(\rightarrow\) OUVERT. Si remt = 1 \(\rightarrow\) STOP_OUVERTURE.
  • Depuis OUVERT : Si remt = 1 \(\rightarrow\) FERMETURE.
  • Depuis FERMETURE : Si sen2 = 1 \(\rightarrow\) FERME. Si remt = 1 \(\rightarrow\) STOP_FERMETURE.
  • Depuis STOP_OUVERTURE : Si remt = 1 \(\rightarrow\) FERMETURE (inversion de sens).
  • Depuis STOP_FERMETURE : Si remt = 1 \(\rightarrow\) OUVERTURE (inversion de sens).

Question 2.2 : Donnez un diagramme de transitions de la machine à états de Moore si le signal remt est un signal continu tant qu'on appuie sur le bouton. Il faut détecter les passages à 0.

Correction et raisonnement

Étape 1 : Identification du problème (Bouton continu).
Si remt reste à '1' sur plusieurs cycles d'horloge, la machine basculerait instantanément entre, par exemple, OUVERTURE et STOP_OUVERTURE à chaque coup d'horloge.

Étape 2 : Résolution avec des états d'attente (Wait states).
Pour éviter le rebouclage indésirable, il faut attendre le relâchement du bouton (remt = 0) avant de valider l'action. On double les états déclenchés par remt :

  • FERME : Si remt = 1 \(\rightarrow\) ATTENTE_RELACHEMENT_OUV (ctr = "0X").
  • ATTENTE_RELACHEMENT_OUV : Si remt = 0 \(\rightarrow\) OUVERTURE. Si remt = 1, boucle sur lui-même.
  • OUVERTURE : Si remt = 1 \(\rightarrow\) ATTENTE_RELACHEMENT_STOP (ctr = "0X").
  • ATTENTE_RELACHEMENT_STOP : Si remt = 0 \(\rightarrow\) STOP_OUVERTURE.

Cette logique s'applique symétriquement pour la fermeture.
