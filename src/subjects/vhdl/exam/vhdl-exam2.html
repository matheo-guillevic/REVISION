Partie 1

Exercice 1 : Conception et validation d'un « timer » [90 points]

Contexte

L'objectif est de concevoir un timer générant périodiquement une impulsion d'une période d'horloge de large . Ce timer est contrôlé par une horloge interne de période 1 \(\mu\)s . Il a une largeur de 32 bits .

Lecture du Schéma : Liste des éléments et leur utilité

Le schéma illustre l'architecture matérielle (chemin de données et contrôle) du timer. Voici comment le lire pas-à-pas :

  • 1 microsec oscillator (Oscillateur) : Fournit le signal d'horloge de base (période de 1 \(\mu\)s) qui cadence tous les éléments synchrones du circuit .
  • Bloc "-1" (Décrémenteur combinatoire) : Réalise une opération de soustraction de 1 sur l'entrée \(M\) . Cela est nécessaire car le compteur décrémente jusqu'à 0 inclus. Pour compter \(M\) cycles, il faut démarrer le comptage à \(M-1\).
  • 32-bit register (Registre de mémorisation) : Sauvegarde la valeur \(M-1\) à chaque fois que la commande load est active (à '1'). Il sert de mémoire pour conserver la configuration du timer (la période souhaitée) de manière autonome, même si le signal externe \(M\) vient à changer par la suite.
  • 4-bit 2x1 Mux (Multiplexeur 2 vers 1) : Aiguille la donnée qui sera chargée dans le compteur. Si load est à '1', il laisse passer directement la nouvelle valeur calculée \(M-1\). Si load est à '0', il laisse passer la valeur sauvegardée dans le registre (pour recharger automatiquement le compteur à chaque fin de cycle).
  • Porte logique OU (OR gate) : Génère le signal de chargement (ld) du compteur. Le compteur est rechargé dans deux situations : soit par un ordre externe de l'utilisateur (load = 1), soit de manière autonome quand le compteur atteint zéro et lève son drapeau tc.
  • 32-bit down-counter (Décompteur 32 bits) : C'est le cœur du timer. Activé par enable, il décrémente sa valeur à chaque front d'horloge. Lorsqu'il atteint 0, sa sortie tc (Terminal Count) passe à 1 . Cela déclenche simultanément l'impulsion de sortie Q vers l'extérieur et le rechargement automatique du compteur via la porte OU pour recommencer un cycle.

Question 1

Énoncé : Quelle valeur doit être programmée (c'est-à-dire envoyée vers M) pour générer une impulsion toutes les 300 ms ?

Correction et Explications :

  • L'horloge a une période de 1 \(\mu\)s .
  • On souhaite une période de 300 ms, soit \(300 \times 10^3 = 300\,000\) \(\mu\)s.
  • Il faut donc compter \(300\,000\) cycles d'horloge entre chaque impulsion.
  • Le circuit décrémente à partir de \(M-1\) jusqu'à \(0\) (soit \(M\) états au total).
  • Réponse : La valeur à programmer sur l'entrée M est donc 300 000.

Question 2

Énoncé : Faire un chronogramme pour deux impulsions avec \(M=3\) puis deux impulsions avec \(M=2\) .

Correction et Explications (Description du chronogramme) : (En LaTeX, un chronogramme complexe est souvent décrit sous forme de table d'états cycle par cycle pour plus de clarté).

  • Cycle 0 (Initialisation) : load = 1, \(M = 3\). Le bloc -1 calcule M32 = 2. Le registre R32 charge 2. Le Mux32 laisse passer M32 (2). Le compteur C charge 2 car la porte OU donne 1 (load=1).
  • Cycle 1 : load = 0, enable = 1. C = 2. tc (Q) = 0. Au front d'horloge, C décrémente à 1.
  • Cycle 2 : C = 1, Q = 0. Au front d'horloge, C décrémente à 0.
  • Cycle 3 (Impulsion) : C = 0, Q = 1 (tc=1 car C=0) . La porte OU vaut 1 (car Q=1). Le Mux32 laisse passer R32 (qui vaut toujours 2). Au front d'horloge, C recharge 2.
  • Cycle 4, 5, 6 : C passe de 2 à 1, puis à 0. On obtient une 2ème impulsion.
  • Cycle 7 (Changement M=2) : On met load = 1, \(M = 2\). M32 = 1. R32 et C chargent 1.
  • Cycle 8 (Impulsion courte) : load = 0, enable = 1. C passe de 1 à 0 au cycle 9. La période est maintenant de 2 coups d'horloge.

Question 3

Énoncé : Est-ce que les périodes des impulsions générées correspondent aux valeurs de M successives (3 puis 2) ? Si ce n'est pas le cas, expliquez .

Correction et Explications : Oui, les périodes correspondent parfaitement. Pour \(M=3\), le compteur prend les valeurs \(2 \rightarrow 1 \rightarrow 0\) (soit 3 cycles d'horloge par impulsion). Pour \(M=2\), il prend les valeurs \(1 \rightarrow 0\) (soit 2 cycles). Le bloc "-1" en amont permet justement de corriger le cycle supplémentaire dû au passage par 0, garantissant que la période réelle en cycles est exactement égale à \(M\).

Question 4

Énoncé : Donnez une description en VHDL de l'entité du timer .

Correction :

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all; -- Requis pour les operations arithmetiques

entity timer is
    port (
        clk    : in  std_logic;

        M      : in  std_logic_vector(31 downto 0);
        Q      : out std_logic
    );
end entity timer;

Question 5

Énoncé : Décrire en VHDL RTL l'architecture rtl de ce circuit .

Correction et Explications : L'architecture doit traduire les blocs de la figure .

architecture rtl of timer is
    -- Signaux internes
    signal M32_uns, R32_uns, Mux32_uns, C_uns : unsigned(31 downto 0);
    signal OR1, Q_int : std_logic;
begin
    -- Instructions concurrentes (Questions 5.3 et 5.4)
    M32_uns <= unsigned(M) - 1; 
    Mux32_uns <= M32_uns when load = '1' else R32_uns;
    
    OR1 <= load or Q_int;
    Q_int <= '1' when C_uns = to_unsigned(0, 32) else '0';
    Q <= Q_int; -- Connexion de la sortie interne vers le port externe

    -- Processus pour le registre R32 (Question 5.1)
    process(clk)
    begin
        if rising_edge(clk) then
            if load = '1' then
                R32_uns <= M32_uns;
            end if;
        end if;
    end process;

    -- Processus pour le decompteur C (Question 5.2)
    process(clk)
    begin
        if rising_edge(clk) then
            if OR1 = '1' then
                C_uns <= Mux32_uns; -- Chargement synchrone
            elsif enable = '1' then
                C_uns <= C_uns - 1; -- Decomptage synchrone
            end if;
        end if;
    end process;

end architecture rtl;

Question 6 (Commentaire Verilog)

Énoncé : Commentez la fonction du code Verilog fourni .

Correction et Explications : Le code décrit un multiplexeur 2 vers 1 de manière combinatoire (always @(load or M32 or R32)). En fonction de la valeur de load, la sortie MUX32 prend la valeur de R32 (si load = 0) ou de M32 (si load = 1) .

Question 7 (Bascules)

Énoncé : Combien de bascules générera la synthèse de votre architecture précédente ? Justifiez .

Correction et Explications : Le circuit générera 64 bascules D. Justification : L'architecture contient deux processus synchrones (sensibles au front d'horloge clk).

  • Le premier mémorise le signal R32, qui est un vecteur de 32 bits \(\rightarrow\) 32 bascules.
  • Le second mémorise le compteur C, qui est également un vecteur de 32 bits \(\rightarrow\) 32 bascules.

Les autres éléments (Mux, bloc -1, porte OU) sont purement combinatoires et ne génèrent pas de bascules.

Question 8 (Testbench)

Énoncé : Donnez une description du testbench tb_timer permettant de retrouver exactement le même chronogramme que celui de la question 2 .

Correction :

library ieee;
use ieee.std_logic_1164.all;
use ieee.numeric_std.all;

entity tb_timer is
end entity tb_timer;

architecture sim of tb_timer is
    signal clk    : std_logic := '0';
    signal load   : std_logic := '0';
    signal enable : std_logic := '0';
    signal M      : std_logic_vector(31 downto 0) := (others => '0');
    signal Q      : std_logic;
    
    constant CLK_PERIOD : time := 1 us;
begin
    -- Instanciation du composant (UUT)
    UUT: entity work.timer
        port map (
            clk => clk, load => load, enable => enable, M => M, Q => Q
        );

    -- Generation de l'horloge
    clk_process : process
    begin
        clk <= '0';
        wait for CLK_PERIOD / 2;
        clk <= '1';
        wait for CLK_PERIOD / 2;
    end process;

    -- Stimuli (Scenario Q2)
    stim_proc : process
    begin
        -- Initialisation (Cycle 0)
        M <= std_logic_vector(to_unsigned(3, 32));
        load <= '1';
        enable <= '0';
        wait for CLK_PERIOD;
        
        -- Comptage normal pour generer deux impulsions
        load <= '0';
        enable <= '1';
        wait for 6 * CLK_PERIOD; -- Attend 6 cycles (2 impulsions de periode 3)
        
        -- Changement de M (M=2)
        M <= std_logic_vector(to_unsigned(2, 32));
        load <= '1';
        wait for CLK_PERIOD;
        
        -- Nouveau comptage pour generer deux impulsions
        load <= '0';
        wait for 4 * CLK_PERIOD; -- Attend 4 cycles (2 impulsions de periode 2)
        
        -- Fin simulation
        enable <= '0';
        wait;
    end process;
end architecture sim;
Partie 2

Exercice 2 : Contrôleur de porte de garage [30 points]

Contexte

Une porte de garage est contrôlée par une machine à états. ctr(1) gère l'alimentation du moteur (1=ON, 0=OFF) et ctr(0) définit le sens (0=ouverture, 1=fermeture) . Les capteurs sen1 et sen2 détectent respectivement l'ouverture totale et la fermeture totale . Le bouton remt donne les ordres .

Question 2.1

Énoncé : Donnez un diagramme de transitions de la machine à états de Moore avec remt sous forme d'impulsion d'une seule période .

Correction et Explications : On modélise le système en définissant les états suivants et leurs sorties associées (ctr(1..0)) :

  • FERME (ctr = "0X") : État initial (après rst) . Si remt = 1, transition vers OUVERTURE .
  • OUVERTURE (ctr = "10") : Le moteur ouvre. Si sen1 = 1, transition vers OUVERT. Si remt = 1, transition vers STOP_OUVERTURE .
  • OUVERT (ctr = "0X") : Si remt = 1, transition vers FERMETURE .
  • FERMETURE (ctr = "11") : Le moteur ferme. Si sen2 = 1, transition vers FERME. Si remt = 1, transition vers STOP_FERMETURE .
  • STOP_OUVERTURE (ctr = "0X") : Arrêt en cours d'ouverture. Si remt = 1, transition vers FERMETURE .
  • STOP_FERMETURE (ctr = "0X") : Arrêt en cours de fermeture. Si remt = 1, transition vers OUVERTURE .

Question 2.2

Énoncé : Donnez un diagramme de transitions de la machine à états de Moore si le signal remt est un signal continu tant qu'on appuie sur le bouton. Il faut détecter les passages à 0 .

Correction et Explications : Puisque remt n'est plus une impulsion d'un cycle, rester dans l'état (par exemple) FERME ferait boucler la machine immédiatement entre OUVERTURE et STOP_OUVERTURE à la vitesse de l'horloge. Pour corriger cela dans une FSM de Moore, il faut doubler les états où l'on s'arrête (ou créer des états transitoires d'attente de relâchement du bouton). Par exemple, pour chaque action déclenchée par remt=1, on ne passe à l'état actif final que lorsque remt=0 :

  • FERME : Si remt = 1, aller vers ATTENTE_RELACHEMENT_OUVERTURE (Moteur = 10, ou 0X selon choix sécuritaire).
  • ATTENTE_RELACHEMENT_OUVERTURE : Si remt = 0, aller vers OUVERTURE. Si remt = 1, on boucle ici.
  • OUVERTURE : Si remt = 1, aller vers ATTENTE_RELACHEMENT_STOP (Moteur OFF).
  • ATTENTE_RELACHEMENT_STOP : Si remt = 0, aller vers STOP_OUVERTURE.

Note : Une méthode plus optimisée industriellement consiste à générer un signal remt_pulse en amont de la machine d'états à l'aide d'un détecteur de front montant synchrone, ce qui permet de conserver exactement la machine de la question 2.1.
